٤ · نظام QR والحضور — المواصفات الكاملة

القلب النابض للنظام — سرعة تسجيل الحضور مع دقة كاملة ومنع أي تلاعب.

🔌

بروتوكول ربط القارئ بالنظام — HID Keyboard Emulation (الموصى به)

✅
الخيار المُختار: USB HID Keyboard Emulation — القارئ يعمل كـ Keyboard ويكتب الـ QR Code مباشرة في حقل Input في المتصفح. لا يحتاج Driver ولا Software. يعمل على أي جهاز كمبيوتر بدون إعداد. يعمل بدون إنترنت (Offline). الأشيع في الجيمات والمحلات. السعر في مصر: 150–500 ج.م.

🟢 HID — كيف يعمل؟

القارئ يمسح QR ← يكتب النص كـ keyboard في input مخصص في المتصفح ← JavaScript يستمع لـ keydown + Enter ← يُرسل QR للـ Backend عبر WebSocket ← الاستجابة لحظية ← يُعيَّد focus للـ input تلقائياً للمسح التالي

🟡 IP Camera — بديل

كاميرا IP تُمسح QR من مسافة — أغلى ثمناً وأعقد في الإعداد. تحتاج مكتبة ZXing أو OpenCV على السيرفر. مناسبة لو الجيم عاوز تجربة "بدون لمس" لكن ليست ضرورية.

🟡 كاميرا الموبايل — بديل

html5-qrcode يفتح كاميرا الهاتف. مناسب كـ backup لو القارئ اتعطل. يُستخدم في التسجيل اليدوي من الموبايل. لكن مش بديل مناسب للحضور السريع.

صورة العضو — ميزة أمان أساسية (مش اختيارية)

فور مسح أي QR، تظهر صورة العضو بوضوح على الشاشة قبل تسجيل الحضور. موظف الاستقبال يتأكد بصرياً. إذا الوجه لا يطابق → يرفض يدوياً. هذا يمنع إعارة الكارت لشخص آخر.

🚨
لماذا هي أمان أساسي: الكارت الورقي أو البلاستيكي سهل الإعارة. بدون التحقق البصري، عضو منتهي يُعطي كارته لصديقه. الصورة + الاسم على الشاشة كبيرة يُصعّل التحايل.
⚙️
التنفيذ: الصورة تُرفع عند إضافة العضو (اختيارية لكن مُوصى بها بشدة) | تُعرض كـ avatar كبير في شاشة الحضور | إذا لا توجد صورة: يظهر Placeholder مع تحذير "لا توجد صورة - تحقق يدوياً"

إدارة مخزون الكروت المادية (QR Cards Inventory)

صفحة مستقلة لتتبع الكروت المادية المُطبَّعة — تمنع الفوضى وتوثّق كل كارت من لحظة طباعته حتى إلغائه.

  • كل كارت له: QR ID + حالة + العضو المرتبط به
  • حالات الكارت: متاح / مُوزَّع / مفقود / ملغي / تالف
  • تقرير: كم كارت مطبوع / موزع / مفقود
  • إشعار للأدمن عند تسجيل كارت مفقود
🔄
Flow استبدال كارت ضائع أو تالف:
① الموظف يفتح ملف العضو
② يضغط "استبدال الكارت"
③ يُدخل سبب الاستبدال (ضائع / تالف)
④ النظام يُلغي QR القديم فوراً (لا يعمل في الحضور)
⑤ يُنشئ QR ID جديد مرتبط بنفس العضو
⑥ يُطبع الكارت الجديد
⑦ Log كامل: من أجرى + متى + السبب
📵

وضع Offline — التفاصيل الكاملة

📦 التخزين المحلي

IndexedDB (مش localStorage) — يتحمل كميات أكبر من البيانات ويعمل بشكل async. الحضور المُسجَّل offline يُحفظ فيه مؤقتاً مع timestamp دقيق.

🔄 المزامنة عند العودة

عند عودة الاتصال: Service Worker يكتشف الـ online event ← يُرسل الحضور المحفوظ للسيرفر ← السيرفر يتحقق منه ويُطبّق نفس قواعد منع التكرار بالـ timestamp الأصلي.

⚠️ Conflict Resolution

لو نفس QR اتسجل offline وonline في نفس الوقت: السيرفر يقارن الـ timestamps — الأول يُقبَل، الثاني يُرفض كـ "تكرار" ويُسجَّل في Log مع ملاحظة "تعارض offline/online".

شاشة الحضور — المنطق الكامل

QR Flow: القارئ يرسل QR → النظام يبحث فوراً → منطق المنع: لو نفس QR اتقرأ خلال أقل من 4 ساعات → يُرفض + تحذير (المدة قابلة للضبط من الإعدادات).

حالات النتائج: ① الاشتراك منتهي → اسم العضو + "الاشتراك منتهي - يلزم التجديد" بلون أحمر | ② الاشتراك نشط → تُخصم حصة + يُسجل الدخول باللون الأخضر | ③ الحصص انتهت → تنبيه "انتهت الحصص" باللون البرتقالي.

التسجيل اليدوي: الموظف يبحث بالاسم من الاشتراكات النشطة + يضيف ملاحظة "تسجيل يدوي - بواسطة [اسم الموظف]" | عرض حضور اليوم: جدول بكل من حضر + الوقت + طريقة التسجيل.

منطق التجميد وتمديد تاريخ الانتهاء — Logic صريح

القاعدة الأساسية: عند تجميد الاشتراك لمدة X يوم، يُمدَّد تاريخ الانتهاء تلقائياً بنفس عدد الأيام.

مثال: اشتراك ينتهي 30 يناير → تجميد 15 يوم ابتداءً من 10 يناير → تاريخ الانتهاء الجديد = 14 فبراير (30 يناير + 15 يوم).

في قاعدة البيانات: frozen_at | frozen_until | freeze_days_count | original_end_date | new_end_date | كل هذا مُسجَّل في Audit Log | لا يمكن للموظف تعديل تاريخ الانتهاء مباشرة — التمديد تلقائي فقط عبر منطق التجميد.

نظام الموافقة على الخصومات — منع التلاعب

المشكلة بدون هذا النظام: موظف استقبال يُعطي خصم 50% لصديقه دون علم الإدارة.

الحل: الاستقبال يكتب المبلغ المقترح + السبب → يُرسَل للأدمن كـ "طلب خصم معلق" → تنبيه Socket.io للأدمن فوراً → Admin يوافق أو يرفض مع ملاحظة → عند الموافقة: يُطبَّق الخصم ويُصدر الإيصال → Log كامل بكل خطوة.

٦ب · إرسال العروض والحملات (Campaigns)

نظام إرسال جماعي ذكي ومنضبط للعروض والتذكيرات — مع استهداف دقيق وحماية من الحظر.

🎯

خطوات إنشاء حملة

  • ① اختيار الفئة المستهدفة: منتهي الاشتراك / نشط / جنس معين / نوع اشتراك معين
  • ② كتابة الرسالة: من الصفر أو اختيار Template جاهزة
  • ③ جدولة الإرسال: فوري أو وقت محدد (تاريخ + ساعة)
  • ④ معاينة قبل الإرسال: عرض عينة للرسالة + عدد المستقبلين
  • ⑤ تقرير نتائج: أُرسلت / فشلت / لم تُرسَل بعد
🛡️

حماية من الحظر في الإرسال الجماعي

  • Delay تلقائي 10–20 ثانية بين كل رسالة
  • لا إرسال قبل الساعة 9 صباحاً أو بعد 9 مساءً
  • حد يومي للرسائل (قابل للضبط من الإعدادات)
  • استخدام Templates معتمدة من Meta مسبقاً
  • توقف تلقائي إذا ظهر خطأ متكرر
  • سجل كامل لكل رسالة مع حالتها
🎉

أنواع الرسائل التلقائية

  • ترحيب عند التسجيل + QR (فوري)
  • تذكير انتهاء الاشتراك (X أيام قبل)
  • إشعار تجديد الاشتراك (عند التجديد)
  • إشعار تجميد الحساب + فك التجميد
  • تهنئة أعياد (مجدولة سنوياً)
  • تنبيه انتهاء الحصص (فور آخر حصة)
📊

سجل الرسائل

  • كل رسالة مسجلة: المستقبل + المحتوى + الوقت
  • الحالة: أُرسلت ✅ / فشلت ❌ / معلقة ⏳
  • سبب الفشل إن وُجد
  • من أرسلها (يدوي / تلقائي)
  • فلتر بالنوع والتاريخ والحالة
  • تصدير السجل PDF/Excel
🔧

API المُوصى به

  • Twilio: موثوق + دعم عربي + سهل التكامل
  • 360dialog: متخصص واتساب بزنس
  • Meta Cloud API: مباشر بدون وسيط
  • Templates يجب اعتمادها من Meta مسبقاً
  • رسائل 1-to-1 بدون مجموعات

٧ب · نظام الشيفتات وتسليم النقدية

نظام صارم لمنع السرقة وضمان تسوية النقدية في كل شيفت — كل ج.م مُحاسَب.

🚨
لماذا هذا النظام مهم جداً؟ بدون نظام شيفتات، الموظف يمكنه استلام أموال غير مسجلة. هذا النظام يُلزم الموظف بتسوية كاملة عند انتهاء شيفته — لا يمكن إغلاق الشيفت قبل مطابقة النقدية.
🔄

دورة حياة الشيفت

  • فتح الشيفت: الموظف يسجل بداية الشيفت + رصيد الخزنة المبدئي
  • أثناء الشيفت: كل اشتراك ودفعة تُسجَّل بالشيفت تلقائياً
  • إغلاق الشيفت: الموظف يُدخل النقدية الموجودة فعلياً في يده
  • المطابقة: النظام يحسب المتوقع ويقارنه بالفعلي
  • إذا في فارق: الموظف يُوضح السبب + الإدارة تُراجع
  • تسليم للإدارة: النقدية تُسلَّم مع تقرير الشيفت
📊

تقرير الشيفت الكامل

  • عدد الاشتراكات التي باعها الموظف
  • عدد الحصص التي سجّلها
  • إجمالي النقدية المستلمة
  • إجمالي الدفع بشبكة/تحويل
  • النقدية المتوقعة vs الفعلية
  • الفارق (إن وُجد) + سببه
  • وقت بداية ونهاية الشيفت
  • توقيع رقمي على التسليم
⚖️

نظام تسليم الخزنة بين الموظفين

① الموظف A يُسلِّم

يُدخل المبلغ الموجود لديه | النظام يُولِّد كود تسليم | يطبع تقرير مختصر للتسليم

② الموظف B يُستلِم

يؤكد المبلغ المُستلَم | إذا في فارق يُسجَّل فوراً | الاثنان يوقعان رقمياً

③ النظام يُسجِّل

السجل محفوظ بالتاريخ والوقت | لا يمكن تعديله | الأدمن يرى كل التسليمات

٨ · نظام الأمان ومنع التلاعب

القوة الحقيقية للنظام — لا مجال لأي تلاعب أو سرقة.

🚫 لا حذف حقيقي أبداً

كل البيانات تستخدم Soft Delete. الحذف ظاهري فقط — البيانات موجودة في قاعدة البيانات وقابلة للاسترجاع.

📋 Log لكل حركة

كل عملية (إضافة/تعديل/حذف ظاهري/تسجيل حضور/دفع) تُسجَّل مع: اسم الموظف + الوقت بالثانية + القيمة قبل وبعد.

🔗 QR مرتبط بعضو واحد

كل QR ID فريد ومرتبط بعضو واحد فقط. لا يمكن استخدامه لعضو آخر. إذا انتهى الاشتراك — QR لا يعمل.

⏱ منع التكرار (45 دقيقة)

النظام يرفض تسجيل الحضور للعضو نفسه قبل مرور 45 دقيقة. المدة قابلة للتعديل من الإعدادات.

💻 تتبع IP والجهاز

كل جلسة مسجلة بـ IP ورقم الجهاز. إذا تغيّر الجهاز يُرسَل تنبيه للأدمن. Session Management قوي.

🔒 صلاحيات دقيقة

كل موظف يرى ويعمل فقط ما يملك صلاحية عليه. لا يوجد "رؤية زائدة". الأدمن يغلق/يفتح صلاحيات فورياً.

١٣ · التحسينات والإضافات المقترحة

📌
يجب ضروري لـ Production | مُوصى به يُحسّن الأمان والكفاءة | مستقبلاً قيمة مضافة

يجب 📸 صورة العضو — أمان أساسي

عند مسح QR تظهر صورة العضو كبيرة فوراً. موظف الاستقبال يتحقق بصرياً. تمنع إعارة الكارت لأشخاص آخرين. إذا لا توجد صورة: تحذير "تحقق يدوياً".

يجب 🔒 نسخ احتياطية تلقائية

Backup يومي تلقائي لـ MySQL. الاحتفاظ بـ 30 يوم. رفع نسخة على Google Drive / S3. تنبيه واتساب للأدمن إذا فشل الـ Backup.

يجب 🧾 إيصال PDF عند كل دفعة

Puppeteer يُولِّد إيصال PDF فور الدفع: اسم + نوع اشتراك + مبلغ بالجنيه المصري + خصم + توقيع رقمي + تاريخ. قابل للطباعة + يُرسَل واتساب تلقائياً.

يجب ✅ نظام موافقة الخصومات

الاستقبال يطلب خصماً + سبب → Admin يُبلَّغ Socket.io → يوافق أو يرفض. لا خصم يُطبَّق بدون Admin. حد مسموح للاستقبال (مثلاً 10%) يُعدَّل من الإعدادات.

يجب 💳 صفحة إدارة مخزون الكروت

تتبع كل كارت: متاح / مُوزَّع / مفقود / ملغي / تالف. Flow استبدال كارت مع إلغاء QR القديم فوراً وإصدار جديد. Log لكل عملية استبدال.

يجب 📵 Offline Mode بـ IndexedDB

حضور يُحفظ محلياً في IndexedDB عند انقطاع الإنترنت. Service Worker يُزامن عند العودة. Conflict Resolution: لو نفس QR اتسجل offline وonline → الأقدم يُقبَل والأحدث يُرفض.

مُوصى به 🔔 Push Notifications للموظفين

Web Push Notifications (PWA) تصل حتى لو المتصفح مغلق. مكمّلة لـ Socket.io (الذي يعمل فقط والمتصفح مفتوح). أحداث مهمة: فارق نقدية في الشيفت / فشل واتساب / اشتراك انتهى الآن.

مُوصى به 📊 Dashboard تفاعلي متقدم

رسوم بيانية Real-time: حضور اليوم بالساعة، أوقات الذروة، أكثر الأعضاء حضوراً، الإيرادات الشهرية بالجنيه المصري.

مُوصى به 📱 تطبيق موبايل للاستقبال

PWA أو React Native يستخدم كاميرا الهاتف كـ QR Scanner. مناسب كـ backup عند تعطل القارئ أو للتسجيل من أي مكان في الجيم.

مُوصى به 🎁 نظام الكوبونات والعروض الموسمية

كوبون بكود + نسبة خصم + صلاحية. عروض رمضان / الصيف بمدة محددة. يُضاف عند إنشاء الاشتراك. Log لكل كوبون استُخدم.

مُوصى به 🔔 تنبيه يومي للأدمن

كل صباح واتساب للأدمن: عدد الاشتراكات التي ستنتهي خلال 3 أيام + إجمالي إيرادات الأمس + أي فوارق نقدية لم تُحسَم.

مُوصى به 🌐 WhatsApp Cloud API الرسمي (مستقبلاً)

whatsapp-web.js غير رسمي وقد يُحجب مع تحديثات واتساب. للتجاري الجدي: Meta Cloud API عبر Twilio أو 360dialog. أكثر استقراراً. يُنصح بالتخطيط للترقية بعد الإطلاق الأولي.

مستقبلاً 📲 تطبيق العضو

تطبيق للعضو: حضوره، حصصه المتبقية، تاريخ انتهاء اشتراكه، إيصالات الدفع. بدون الحاجة لتواصل مع الاستقبال.

مستقبلاً 🤖 AI توقع الانقطاع

تحليل نمط الحضور لكل عضو. إذا الحضور انخفض فجأة → رسالة تحفيز تلقائية.

مستقبلاً 📅 حجز الحصص مسبقاً

العضو يحجز مكانه في الحصة عبر واتساب أو رابط. تأكيد تلقائي. عداد الأماكن في لوحة الإدارة.

مستقبلاً 🏆 نقاط وحوافز الالتزام

نقطة لكل حضور. 30 نقطة = حصة مجانية أو خصم. يُعرض في بروفايل العضو ويُحفّز الالتزام.
